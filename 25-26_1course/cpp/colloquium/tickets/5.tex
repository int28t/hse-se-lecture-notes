\section{Функциональная декомпозиция, ссылки и константность}

\subsection{Темы}
Материал этого раздела преимущественно рассматривается в рамках лекции 5. Включает:
\begin{itemize}
  \item Понятие декомпозиции задач и функции как основной строительный блок программы.
  \item Функциональная декомпозиция в C++: свободные функции и методы, интерфейс/реализация.
  \item \texttt{void} как неполный тип; \texttt{void*}.
  \item Определение функции: заголовок, сигнатура, формальные параметры, тело, \texttt{return}.
  \item Caller / callee / prototype.
  \item Признаки хорошей функции, smell-коды, принципы декомпозиции.
  \item Принципы S.O.L.I.D. — кратко с примерами.
  \item Параметры функций: by-value, by-reference, const-ref, ссылки-результаты, правило «Правило №5».
  \item CV-квалификаторы: \texttt{const}, \texttt{volatile}.
  \item Ссылки: синтаксис, binding, нельзя перепривязать, идиомы (range-based for).
  \item Идиоматические правила передачи аргументов (мелкие/большие объекты).
\end{itemize}

\subsection{Контрольные вопросы и ответы}

\subsubsection{Как решить большую задачу так, чтобы решение было корректным, проверяемым, воспроизводимым, переиспользуемым, читаемым?}
Основные практики:
\begin{itemize}
  \item Разбить задачу на маленькие независимые функции (функциональная декомпозиция).
  \item Для каждой функции формализовать интерфейс (что принимает, что возвращает, побочные эффекты).
  \item Писать юнит-тесты для каждой функции (изолированное тестирование).
  \item Использовать инкапсуляцию и модули для управления зависимостями.
  \item Документировать предусловия/постусловия/инварианты.
  \item Следовать SOLID-подходам и чистым методикам разработки (одна ответственность, малые модули).
\end{itemize}

\subsubsection{Что такое функция с точки зрения языка C++?}
Функция --- именованный блок инструкций, принимающий параметры и возвращающий (возможно) значение. Формально состоит из заголовка (имя, список формальных параметров, возвращаемый тип), опциональных спецификаторов и тела (блок инструкций). Функция реализует отношение «вход---выход» и может иметь побочные эффекты.

\subsubsection{Как определить функцию / процедура? Какие основные компоненты функции?}
Определение включает:
\begin{itemize}
  \item возвращаемый тип (может быть \texttt{void}),
  \item имя функции,
  \item список формальных параметров \texttt{(type name, ...)} — может быть пуст,
  \item (опционально) квалификаторы (\texttt{const}, \texttt{noexcept}, \texttt{constexpr}, \texttt{inline}, спецификаторы видимости и т.д.),
  \item тело --- блок \texttt{\{ ... \}} с инструкциями,
  \item инструкция \texttt{return} для возврата значения (если тип не \texttt{void}).
\end{itemize}

Пример:
\begin{verbatim}
int sum(int a, int b) {
    return a + b;
}
\end{verbatim}

\subsubsection{Что такое неполный тип \texttt{void}? Можно ли породить объект такого типа? А типа \texttt{void*}?}
\begin{itemize}
  \item \texttt{void} --- неполный тип, означающий «нет значения»; нельзя создавать объект типа \texttt{void} (неопределённая память/безразмерный тип).
  \item \texttt{void*} --- указатель на неизвестный тип; можно создавать, хранить и преобразовывать (с приведениями) указатели \texttt{void*}. \texttt{void*} полезен для хранения "сырых" адресов, но операция разыменования невозможна без приведения к конкретному типу.
\end{itemize}

\subsubsection{Что такое свободная функция и что такое функция-член? Примеры.}
\begin{itemize}
  \item \textbf{Свободная (non-member) функция} --- функция, не являющаяся частью класса/структуры: \texttt{int add(int a, int b);} . Хороша для операций, не требующих доступа к внутреннему состоянию объекта.
  \item \textbf{Функция-член (member)} --- определена внутри класса: \texttt{struct S \{ int f(); \};}. Имеет неявный параметр \texttt{this}.
\end{itemize}
Пример \texttt{std::getline} --- свободная функция в пространстве имён \texttt{std} (она принимает \texttt{std::istream\&} в качестве первого аргумента).

\subsubsection{Является ли \texttt{std::getline} свободной или членом?}
\texttt{std::getline} --- свободная (non-member) функция: она определена как перегруженная функция в \texttt{namespace std} и принимает \texttt{std::istream\&} и \texttt{std::string\&}.

\subsubsection{Является ли \texttt{static} функция, объявленная в классе, свободной? Как вызвать?}
Статическая функция-член класса \texttt{static} \texttt{bar()} всё равно является \emph{функцией-членом} (belongs to class scope) но не имеет неявного \texttt{this}. Вызывать можно как \texttt{Foo::bar();} или через объект \texttt{f.bar()} (но предпочтительна нотация с классом).

\subsubsection{Что такое side-effect у функции и почему это нежелательно для свободных функций?}
\textbf{Side effect} --- изменение состояния вне локальной области видимости функции (изменение глобальных переменных, I/O, модификация параметров через ссылки). Для свободных функций side effects уменьшают переиспользуемость и тестируемость. Хорошая свободная функция должна быть по возможности без побочных эффектов (функция должна быть детерминированной и позволять композицию).

\subsubsection{Что такое методы для классов и пример метода \texttt{std::string}?}
Метод --- функция, объявленная внутри класса; имеет доступ к \texttt{this} и к членам класса. Пример: \texttt{std::string::length()} — метод, возвращающий длину строки.

\subsubsection{Что такое список формальных параметров функции? Можно ли составить список только из типов?}
Список формальных параметров --- это список пар «тип имя» или только типов (имя можно опустить в объявлении). В объявлении/прототипе можно писать только типы:
\begin{verbatim}
int f(int, double); // допустимо — имена параметров опущены
\end{verbatim}
Но в определении обычно указывают имена для использования внутри тела.

\subsubsection{Что такое сигнатура функции? Зачем её отделять от заголовка и прототипа?}
Сигнатура — набор информации, по которой компилятор различает функции для перегрузки: имя функции + количество и типы (и cv/ref- квалификаторы для методов) формальных параметров. Возвращаемый тип \emph{не} входит в сигнатуру для перегрузки функций. Отдельно рассматривают заголовок/прототип, поскольку сигнатура — это часть интерфейса, а заголовок может содержать также квалификаторы (\texttt{noexcept}, \texttt{constexpr}, атрибуты) и документацию.

\subsubsection{Чем различаются заголовок функции и прототип? Что относится к интерфейсной части?}
\begin{itemize}
  \item \textbf{Прототип} (declaration) --- объявление функции (ее интерфейса): тип возвращаемого значения, имя, параметры; не содержит тела.
  \item \textbf{Заголовок} --- чаще используются как синоним прототипа; но в широком смысле заголовок функции --- верхняя строка определения вместе с её спецификаторами.
  \item Интерфейсная часть = прототип + документация предусловий/постусловий.
\end{itemize}

\subsubsection{Может ли функция иметь более одной точки выхода? Хороший ли это стиль? Может ли \texttt{void} функция иметь \texttt{return}?}
\begin{itemize}
  \item Функция может иметь несколько инструкций \texttt{return}. Это допустимо; стиль — предмет соглашений. В большинстве стилей наличие нескольких точек выхода упрощает ранний выход при ошибках (guard clauses), но может усложнить анализ состояния при cleanup — в C++ RAII решает проблему освобождения ресурсов.
  \item \texttt{void} функция может иметь \texttt{return;} без выражения — для раннего выхода.
\end{itemize}

\subsubsection{Нужно ли компилятору полное определение функции? А линкеру?}
\begin{itemize}
  \item Для компиляции вызова функции обычно достаточно прототипа (описания) — компилятору нужно знать сигнатуру, чтобы корректно сгенерировать код вызова.
  \item Линкер требует полного определения (definition) функции, если функция действительно вызывается и не является inline/templated/constexpr с видимой реализацией в момент компоновки.
\end{itemize}

\subsubsection{Как обычно распределяется интерфейс и реализация по заголовкам и модулем трансляции?}
Интерфейсы объявляются в заголовочных файлах (\texttt{.h/.hpp}) или в модульных интерфейсных единицах; реализации — в \texttt{.cpp} или модульных реализациях. Клиенты включают заголовки или импортируют модули, а линковка обеспечивает объединение реализаций.

\subsubsection{Признаки хорошей функции (и объяснение)} 
Признаки:
\begin{itemize}
  \item \textbf{Единая ответственность} --- делает только одно логическое действие.
  \item \textbf{Короткая} --- небольшое количество строк; легко осмысляется целиком.
  \item \textbf{Понятный интерфейс} --- минимальное количество аргументов; аргументы понятны по именам/типам.
  \item \textbf{Отсутствие неожиданных побочных эффектов} --- побочные эффекты документированы и минимальны.
  \item \textbf{Реентерабельность / чистота} --- если возможно, не использовать глобальное состояние.
  \item \textbf{Хорошая тестируемость} --- можно протестировать в изоляции.
\end{itemize}

\subsubsection{Симптомы плохой функции (smell code)}
\begin{itemize}
  \item Слишком длинная функция (много строк, много уровней вложенности).
  \item Слишком много параметров (>3--4).
  \item Множество побочных эффектов и скрытых зависимостей.
  \item Дублирование кода внутри функции.
  \item Сильная связанность с внешними модулями (затрудняет тестирование).
\end{itemize}

\subsubsection{Что такое реентерабельная функция? Пример параметризуемой и непараметризуемой функции}
\begin{itemize}
  \item \textbf{Реентерабельность} --- функция может быть прервана и затем повторно вызвана параллельно/последовательно без нарушения общей корректности (не использует модульное глобальное изменяемое состояние).
  \item \textbf{Параметризуемость} --- функция имеет параметры, которые определяют поведение. Пример непараметризуемой хорошей функции: \texttt{int digits\_in\_decimal(int n);} — простая, конкретная задача, легко тестируется.
\end{itemize}

\subsubsection{Принцип высокой/низкой связности (coupling \& cohesion)}
\begin{itemize}
  \item \textbf{Когезия (cohesion)} --- степень того, насколько элементы модуля/функции логически связаны. Высокая когезия желательна.
  \item \textbf{Связность (coupling)} --- степень зависимости между модулями. Низкая связность желательна.
\end{itemize}

\subsubsection{Тестируемость функции}
Тестируемая функция:
\begin{itemize}
  \item небольшой интерфейс,
  \item детерминированна (или поведение тестируемо при контролируемых входах),
  \item не зависит от глобальных состояний (или позволяет их подменять).
\end{itemize}

\subsubsection{Оптимальный размер функции и ясность}
Оптимальный размер — такой, чтобы функция выполняла единичную логическую задачу и помещалась в поле зрения разработчика (один экран/несколько экранов). Слишком маленькие функции приводят к росту числа абстракций; слишком большие — к потере ясности.

\subsubsection{Принципы функциональной декомпозиции}
\begin{itemize}
  \item Делить проблему на подзадачи с минимальными взаимными зависимостями.
  \item Выделять интерфейсы и инварианты на границах модулей.
  \item Делать функции выразительными: имя и сигнатура должны объяснять призвание функции.
\end{itemize}

\subsubsection{Принципы S.O.L.I.D. и примеры}
\begin{itemize}
  \item \textbf{S (Single Responsibility)}: класс/функция — одна причина для изменения. Пример: класс \texttt{FileReader} только читает файлы; логика парсинга вынесена в отдельный класс.
  \item \textbf{O (Open/Closed)}: сущности открыты для расширения, закрыты для модификации. Пример: использование виртуального интерфейса для добавления поведения без изменения существующего кода.
  \item \textbf{L (Liskov Substitution)}: подкласс должен быть взаимозаменяем с базовым классом. Пример: подкласс не должен усиливать предусловия и не должен ослаблять постусловия.
  \item \textbf{I (Interface Segregation)}: мелкие специализированные интерфейсы лучше, чем один большой. Пример: разделение интерфейса \texttt{IPrinter} на \texttt{ITextPrinter} и \texttt{IImagePrinter}.
  \item \textbf{D (Dependency Inversion)}: зависеть от абстракций, а не от конкретных реализаций. Пример: передавать \texttt{ILogger\&} в конструкторе вместо использования конкретного \texttt{ConsoleLogger}.
\end{itemize}

\subsubsection{Чем отличаются фактические параметры (arguments) от формальных параметров?}
\begin{itemize}
  \item \textbf{Формальные параметры} --- имена/типы, объявленные в заголовке функции.
  \item \textbf{Фактические параметры (arguments)} --- значения/выражения, передаваемые при вызове.
\end{itemize}

\subsubsection{Что такое перегрузка функций и условия для перегрузок?}
Перегрузка --- наличие нескольких функций с одним именем, различающихся сигнатурами (количество/типы параметров, cv/ref- квалификаторы для методов). Возвращаемый тип не отличает перегрузки. Пример:
\begin{verbatim}
void f(int);
void f(double);
\end{verbatim}

\subsubsection{Методы графического представления взаимодействия функций}
\begin{itemize}
  \item \textbf{Call graph} (граф вызовов) --- вершины = функции, ребра = вызовы.
  \item \textbf{Sequence diagram (UML)} --- временная диаграмма взаимодействия объектов/функций.
  \item \textbf{Activity diagram} --- поток управления.
\end{itemize}

\subsubsection{Что такое CV-квалификаторы? Что такое \texttt{const} и \texttt{volatile}?}
\begin{itemize}
  \item \textbf{CV-квалификаторы} --- \texttt{const} и \texttt{volatile} (часто вместе с \texttt{restrict} в других языках).
  \item \texttt{const} запрещает изменение объекта через данную ссылку/имя; влияет на семантику операций (методы \texttt{const}).
  \item \texttt{volatile} указывает компилятору, что объект может изменяться внешне (аппаратно или потоками) — компилятор должен избегать оптимизаций, предполагающих неизменность.
  \item \texttt{const} используется гораздо чаще; \texttt{volatile} редко (низкоуровневое программирование, драйверы).
\end{itemize}

\subsubsection{Что такое константный объект (const)? Есть ли у него имя, адрес, тип, значение? Что меняет \texttt{const}?}
Константный объект имеет имя, адрес (если lvalue), тип и значение. \texttt{const} запрещает операции, меняющие значение через этот идентификатор; не влияет на представление объекта в памяти, но влияет на операции, доступные в языке (компилятор контролирует модификации).

\subsubsection{Что такое ссылка на объект и зачем она нужна? Как задаётся?}
Ссылка --- альтернативное имя для существующего объекта. Синтаксис: \texttt{T\& r = obj;} . Ссылка не хранит адрес как значение в обычном понимании, и её нельзя перепривязать. Ссылки используются для:
\begin{itemize}
  \item передачи больших объектов без копирования (\texttt{const T\&}),
  \item изменения объекта в вызывающем коде (неконстантная ссылка),
  \item полиморфизма (передача базового по ссылке).
\end{itemize}

Пример binding:
\begin{verbatim}
int x = 42;
int &r = x; // r и x — одно и то же имя (alias)
\end{verbatim}

\subsubsection{Почему говорят, что ссылка --- «еще одно имя для объекта»? Примеры.}
Если у объекта есть несколько имён (переменная, ссылка, указатель, элемент контейнера), они ссылаются на одну и ту же область памяти. Примеры:
\begin{itemize}
  \item \texttt{int x; int \&r = x;} --- изменение через \texttt{r} изменяет \texttt{x}.
  \item range-based for с ссылкой: \texttt{for (auto \&e : vec) e += 1;} — меняет элементы в контейнере.
\end{itemize}

\subsubsection{Можно ли перепривязать ссылку? Отложенное привязывание?}
\begin{itemize}
  \item \textbf{Перепривязать ссылку нельзя.} Ссылка привязывается к объекту при инициализации и далее остаётся с этим объектом (под капотом реализовано через указатель, но семантически — неизменяемое alias).
  \item \textbf{Отложенное привязывание} не поддерживается для обычных ссылок. Альтернативы: указатели (\texttt{T*}) или \texttt{std::reference\_wrapper<T>} / \texttt{std::optional<std::reference\_wrapper<T>>}.
\end{itemize}

\subsubsection{Когда параметры передаются по ссылке? Чем отличаются ссылки от передачи по значению?}
\begin{itemize}
  \item \textbf{By value} (по значению) --- копируется объект (безопаснее, независимее).
  \item \textbf{By reference} (по ссылке) --- передаётся alias; нет копирования; callee может изменить объект в caller'е.
  \item Используют ссылки, чтобы избежать копирования больших объектов или чтобы позволить функции менять аргумент.
\end{itemize}

\subsubsection{Как вернуть из функции более одного значения (не используя массивы/структуры)?}
Варианты:
\begin{itemize}
  \item Использовать выходные параметры по ссылке/указателю:
\begin{verbatim}
void f(int input, int &out1, double &out2) { ... }
\end{verbatim}
  \item Возвращать \texttt{std::pair} / \texttt{std::tuple} / пользовательскую структуру.
  \item В C++17 можно использовать structured bindings: \texttt{auto [a,b] = f();}.
\end{itemize}

\subsubsection{Что такое «Правило № 5»: «В C++ по умолчанию --- всё по значению»?}
Это практическое наблюдение: при отсутствии явного \texttt{\&} или указателя аргументы и возвраты копируются по значению. Чтобы избежать ненужных копий, используют:
\begin{itemize}
  \item \texttt{const T\&} для передачи больших объектов на чтение,
  \item \texttt{T\&} для модификации,
  \item \texttt{T\&\&} и \texttt{std::move} для перемещения/может оптимизировать копии.
\end{itemize}

\subsubsection{Что такое константная ссылка и когда её использовать?}
\texttt{const T\&} --- ссылка, которая не позволяет изменять объект. Используют когда нужно передать большой объект для чтения без копирования:
\begin{verbatim}
void print(const std::string &s) { std::cout << s; }
\end{verbatim}
Если объект мал (например, \texttt{int}, \texttt{double}), копирование дешевле и предпочтительнее.

\subsubsection{Пример, где константная ссылка эффективна; когда предпочесть копирование}
\begin{itemize}
  \item Эффективна: \texttt{void process(const std::vector<std::string>\& v);} — избегает копирования большого вектора.
  \item Копирование предпочтительнее: \texttt{void inc(int x);} — копирование \texttt{int} дешевле, чем использование ссылки.
\end{itemize}

\subsubsection{Правило трёх указателей (ПТУ) — «большие» и «малые» объекты}
Неформальная эвристика: если тип по размерам больше примерно 2--3 указателей (\texttt{sizeof(T) > 3 * sizeof(void*)}), считать его «большим» и передавать по \texttt{const T\&}. Для типов меньшего размера (скалярные) передавать по значению.

\subsubsection{Примеры: когда передавать по значению / по неконстантной ссылке / по константной ссылке}
\begin{itemize}
  \item По значению: \texttt{int}, \texttt{double}, небольшие POD-структуры.
  \item По неконстантной ссылке: \texttt{void update(Point \&p);} --- когда нужно изменить аргумент и вернуть изменения в caller.
  \item По константной ссылке: \texttt{void print(const std::string \&s);} --- большой объект, только чтение.
\end{itemize}

\subsection{Короткий справочник-резюме (чтобы ничего не забыть)}
\begin{itemize}
  \item Прототип в заголовке, определение в \texttt{.cpp} (или экспортируемый модуль).
  \item Компилятор нужен только интерфейс; линкер --- реализация.
  \item Используй \texttt{const T\&} для больших объектов на чтение; \texttt{T} для малых; \texttt{T\&} для изменения.
  \item Ссылки нельзя перепривязать; указатели можно.
  \item \texttt{void} --- неполный тип; \texttt{void*} хранит адрес.
  \item SOLID + маленькие функции + низкая связность = легко тестируемый код.
\end{itemize}
