\section{Основы C++}

\subsection{Темы}
Материал этого раздела преимущественно рассматривается в рамках лекций 2--3. Включает:
\begin{itemize}
  \item Препроцессор языка C(++): назначения, этапы работы в жизненном цикле построения программы, директивы.
  \item Применение препроцессора для конфигурирования текста программы для конфигураций \texttt{debug} и \texttt{release}. Пример с \texttt{cassert}.
  \item «Листинг» программы как вывод препроцессора, содержание.
  \item Директивы условного включения: \texttt{\#if}, \texttt{\#ifdef}, \texttt{\#ifndef}, \texttt{\#else}, \texttt{\#elif}, \texttt{\#endif}.
  \item Директива включения заголовочного файла \texttt{\#include}. Системные (библиотечные) и пользовательские (проектные) заголовки, разница в синтаксисе. Порядок включения заголовков (стилевая договорённость).
  \item Проблема повторного (косвенного) включения заголовочного файла.
  \item Header guard и его классическая реализация. Правила именования header guard с привязкой к пути к файлу и имени проекта. Реализация с помощью \texttt{\#pragma once}.
  \item Заголовочный файл модуля как интерфейсная часть модуля.
  \item Принцип описания (declaration) и определения (definition). Связь описания и определения с интерфейсом и реализацией.
\end{itemize}

\subsection{Контрольные вопросы и развёрнутые ответы}

\subsubsection{Каково назначение препроцессора языка C(++)?}
Препроцессор выполняет текстовые преобразования исходного кода до этапа компиляции. Его назначение:
\begin{itemize}
  \item Подстановка содержимого файлы при директиве \texttt{\#include}.
  \item Управление условной компиляцией (\texttt{\#if}/\texttt{\#ifdef}/\texttt{\#else}...).
  \item Подстановка макросов (\texttt{\#define}).
  \item Генерация конфигурационных вариантов исходного текста (напр., \texttt{debug} vs \texttt{release}).
\end{itemize}
Важно: препроцессор оперирует текстом, а не семантикой языка C++; для него синтаксис и семантика C++ не имеют принципиального значения — он выполняет простые текстовые трансформации.

\subsubsection{С какими сущностями манипулирует препроцессор? Имеет ли для него принципиальное значение синтаксис и семантика языка C++?}
Препроцессор манипулирует:
\begin{itemize}
  \item макросами: \texttt{\#define NAME replacement},
  \item условной компиляцией: \texttt{\#if}, \texttt{\#ifdef}, \texttt{\#ifndef}, \texttt{\#elif}, \texttt{\#else}, \texttt{\#endif},
  \item включениями файлов: \texttt{\#include},
  \item директивами типа \texttt{\#error}, \texttt{\#pragma}.
\end{itemize}
Препроцессор не проверяет семантику C++ — он лишь подготавливает текст, который потом парсит компилятор. Поэтому конструкции препроцессора могут нарушить синтаксис C++ если их неправильно использовать (например, незакрытые \texttt{\#if}).

\subsubsection{На примере директивы \texttt{assert} покажите, как препроцессор меняет код программы для конфигураций сборки \texttt{debug} и \texttt{release}.}
Стандартная макрос-функция \texttt{assert} из \texttt{<cassert>} определяется примерно так: если \texttt{NDEBUG} не определён, \texttt{assert(expr)} проверяет выражение, иначе — превращается в пустой макрос.

Пример кода:
\begin{verbatim}
#include <cassert>

void f(int x) {
    assert(x != 0);
    // ...
}
\end{verbatim}

При сборке в \texttt{debug} (по умолчанию \texttt{NDEBUG} не определён), макрос \texttt{assert} оставляет проверку, и в случае \texttt{x == 0} программа будет аварийно завершена с диагностикой.  
При сборке в \texttt{release} часто определяют \texttt{NDEBUG} (например, через флаг компилятора \texttt{-DNDEBUG}), и тогда макрос \texttt{assert} разворачивается в пустой код — проверка полностью исключается препроцессором и не влияет на бинарник.

Иллюстрация (что делает препроцессор): команда
\begin{verbatim}
g++ -E file.cpp -o file.i
\end{verbatim}
создаст префиксированный листинг \texttt{file.i} с результатом подстановок, где видна либо проверка, либо её отсутствие в зависимости от \texttt{NDEBUG}.

\subsubsection{Что содержит и что не содержит листинг (результат работы препроцессора)?}
\textbf{Листинг (output of preprocessor)} содержит:
\begin{itemize}
  \item исходный код с подставленными содержимыми включённых файлов,
  \item развёрнутые макросы (текст замены),
  \item удалённые комментарии (обычно),
  \item результаты условной компиляции (фрагменты, оставшиеся после \texttt{\#if/\#endif}).
\end{itemize}
\textbf{Листинг не содержит}:
\begin{itemize}
  \item семантической информации компилятора (типов, связываний) — это задача компилятора;
  \item исходных комментариев (они обычно удаляются препроцессором);
  \item макросов как таковых — они развёрнуты (заменены).
\end{itemize}

\subsubsection{Приведите пример конфигурирования текста исходного кода программы условными директивами препроцессора (напр., включения фрагмента кода, релевантного определённой ОС).}
Пример условного включения кода для Windows vs POSIX:
\begin{verbatim}
#ifdef _WIN32
    // код, специфичный для Windows
    #include <windows.h>
#elif defined(__linux__)
    // код, специфичный для Linux
    #include <unistd.h>
#else
    // код для прочих платформ
#endif
\end{verbatim}
Такой подход позволяет поддерживать кроссплатформенный исходник, выбирая нужные куски кода в зависимости от предопределённых макросов компилятора.

\subsubsection{Как включить в модуль системный заголовочный файл? — и заголовочный файл из проекта?}
\begin{itemize}
  \item Системный (библиотечный) заголовок: \verb|#include <vector>|. Компилятор ищет такие файлы в системных/установленных каталогах.
  \item Проектный заголовок: \verb|#include "foo.h"|. Сначала ищется относительно каталога исходного файла, затем в системных каталогах (поведение зависит от компилятора/флагов).
\end{itemize}
Стилевая договорённость: сначала включать стандартные/системные заголовки, затем сторонние (third-party), затем проектные заголовки — чтобы избежать скрытых зависимостей и гарантировать, что заголовки проекта сами корректно включают необходимые зависимости.

\subsubsection{В чем заключается проблема повторного (косвенного) включения заголовочного файла? Приведите пример.}
Проблема: если один и тот же заголовочный файл включается несколько раз в пределах одной единицы трансляции (прямо или косвенно через другие заголовки), то без защиты это приведёт к множественным определениям (например, повторному объявлению классов, перегрузкам define и т.п.), что вызывает ошибки компиляции (множественное определение или синтаксические ошибки).

Пример:
\begin{verbatim}
// a.h
struct A { int x; };

// b.h
#include "a.h"

// c.cpp
#include "a.h"
#include "b.h" // a.h будет включён повторно без защиты -> ошибки
\end{verbatim}

\subsubsection{Что такое header guard и какую проблему он призван предотвратить? Приведите конкретный пример.}
\textbf{Header guard} --- условная компиляция вокруг содержимого заголовочного файла, предотвращающая его повторное включение.

Классическая реализация:
\begin{verbatim}
// foo.h
#ifndef PROJECT_SUBDIR_FOO_H
#define PROJECT_SUBDIR_FOO_H

struct Foo { int x; };

#endif // PROJECT_SUBDIR_FOO_H
\end{verbatim}

Если \texttt{foo.h} включён несколько раз, первый проход определяет макрос \texttt{PROJECT\_SUBDIR\_FOO\_H}, а последующие проходы содержимое между \texttt{\#ifndef} и \texttt{\#endif} пропускают — повторное объявление не происходит.

\subsubsection{Как правильно именовать header guard (стилевые договорённости)?}
Рекомендации:
\begin{itemize}
  \item Включать в имя guard путь к файлу и имя проекта, переводя символы пути в подчёркивания и приводя к верхнему регистру, например: \texttt{MYPROJ\_MODULE\_SUBMODULE\_FOO\_H}.
  \item Это снижает риск коллизий между одинаковыми именами файлов в разных частях проекта или в подключённых библиотеках.
\end{itemize}

\subsubsection{Реализация с помощью \texttt{\#pragma once}}
Альтернатива header guards:
\begin{verbatim}
// foo.h
#pragma once

struct Foo { int x; };
\end{verbatim}
\texttt{\#pragma once} проще и короче; поддерживается всеми современными компиляторами, но формально это не часть стандарта C++ (хотя де-факто повсеместно доступно). Некоторые проекты используют и то и другое для максимальной совместимости.

\subsubsection{Можно ли использовать заголовочный файл без header guard и если да, то какие ограничения это накладывает?}
Технически --- можно, но тогда:
\begin{itemize}
  \item файл не должен содержать определений, которые при множественном включении приведут к ошибкам (например, глобальные переменные с выделением памяти, определения классов без \texttt{inline} и т.д.);
  \item обычно это означает, что такой заголовок может содержать только \emph{inline} функции, шаблоны (которые допускают множественное включение), константы с внутренней связностью (например, \texttt{static const}) и т.п.
\end{itemize}
Практически всегда рекомендуется использовать header guards или \texttt{\#pragma once}.

\subsubsection{Что обычно кладут в заголовочный файл — интерфейсную часть некоторого модуля \texttt{foo}? А что обычно не кладут — и почему?}
\textbf{Обычно кладут в \texttt{foo.h}:}
\begin{itemize}
  \item объявления классов/структур и их публичных интерфейсов,
  \item прототипы (declarations) функций,
  \item объявления глобальных констант/типов (через \texttt{extern} при необходимости),
  \item inline-функции и шаблоны (которые требуют видимости при инстанциации).
\end{itemize}

\textbf{Обычно не кладут:}
\begin{itemize}
  \item большие реализации (тяжёлые определения функций, если они не \texttt{inline}), чтобы не увеличивать время компиляции и избежать множественных определений;
  \item определения глобальных переменных (за исключением \texttt{inline} variables в C++17) — вместо этого используют \texttt{extern} в заголовке и определение в \texttt{.cpp}.
\end{itemize}

\subsubsection{Чем в C++ описание (declaration) принципиально отличается от определения (definition)? Как можно наблюдать эффект описания и эффект определения?}
\textbf{Описание (declaration)} сообщает компилятору о существовании сущности: её имени и типа (например, прототип функции, объявление \texttt{extern int x;}). Описание позволяет компилятору проверить корректность вызовов и ссылок на сущность без знания её реализации.

\textbf{Определение (definition)} создаёт саму сущность: выделяет память для объекта или предоставляет тело функции (например, \texttt{int x = 0;} или \texttt{int f() { return 42; }}).

\textbf{Наблюдение эффектов:}
\begin{itemize}
  \item Если есть только описание (прототип) и отсутствует определение при линковке вызова — линковщик выдаст ошибку undefined reference.
  \item Если есть определение, но код ссылается на него правильно — всё линкуется успешно.
  \item Компилятор требует описания для корректной генерации кода вызова; линкер требует определения для разрешения символов.
\end{itemize}

\subsection{Короткие примеры (безопасные для вставки в \texttt{.tex})}

\subsubsection*{Пример использования условной компиляции для \texttt{assert}}
\begin{verbatim}
#include <cassert>

void f(int x) {
    assert(x != 0); // в релизе при -DNDEBUG эта проверка исчезает
}
\end{verbatim}

\subsubsection*{Пример header guard и pragma once}
\begin{verbatim}
// include/project/sub/foo.h
#ifndef PROJECT_SUB_FOO_H
#define PROJECT_SUB_FOO_H

struct Foo { int x; };

#endif // PROJECT_SUB_FOO_H

// или кратко:
#pragma once
\end{verbatim}

\subsubsection*{Пример порядка include (рекомендация)}
\begin{verbatim}
// recommended order in .cpp:
#include <iostream>   // стандартные заголовки
#include <vector>     // ...
#include "thirdparty/foo.h" // сторонние библиотеки
#include "project/sub/foo.h" // собственные заголовки
\end{verbatim}

\subsection{Итог и рекомендации для экзамена / коллоквиума}
\begin{itemize}
  \item Уметь объяснить, что делает препроцессор и привести пример условной компиляции (например, \texttt{NDEBUG} + \texttt{assert}).
  \item Понимать, что листинг препроцессора (\texttt{g++ -E}) показывает развёрнутый исходный текст.
  \item Уметь показать классический header guard и объяснить, почему имя guard придумывают с привязкой к пути и проекту.
  \item Знать разницу между \texttt{\#include <...>} и \texttt{\#include "..."} и рекомендованный порядок include.
  \item Чётко различать declaration и definition — где они появляются (заголовок vs \texttt{.cpp}) и чего требует компилятор, а чего --- линкер.
\end{itemize}
