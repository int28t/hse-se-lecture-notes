\section{Вокруг языка C++}

\subsection{Темы}
Материал этого раздела даёт обзор экосистемы вокруг языка C++ и включает следующие направления:
\begin{itemize}
  \item Существующие стандарты и драфты языка, жизненный цикл стандартов.
  \item The \textbf{Zero Overhead Principle} --- принцип нулевых накладных расходов и примеры его проявления в коде.
  \item Сборка и организация проектов: CMake --- ключевые директивы и типичные сценарии.
  \item Ключевой принцип: \emph{интерфейсы} и \emph{реализации} (разделение обязанностей).
  \item Заголовочные файлы и модули трансляции: назначение, типичное содержимое и ограничения.
  \item Этапы построения программы (lifecycle): от исходников до исполняемого файла.
  \item Правила именования файлов (Правило № 0) и договорённости по стилю оформления кода.
\end{itemize}

\subsection{Контрольные вопросы}

\subsubsection{Какие стандарты языка существуют? Каков цикл выпуска новых стандартов?}
\textbf{Кратко:} язык C++ развивается стандартами, каждый стандарт --- версия языка. Основные стандарты (хронология): C++98, C++03, C++11, C++14, C++17, C++20, C++23. Разрабатывает стандарт ISO WG21 (ISO/IEC JTC1/SC22/WG21), включая рабочие драфты и предложения.

\textbf{Цикл выпуска:} в последние годы крупные стандарты появляются примерно раз в 2--3 года (C++11, C++14, C++17, C++20, C++23). Между крупными версиями публикуются рабочие драфты и технические спецификации.

\textbf{Примечание:} между стандартами публикуются \emph{Technical Specifications (TS)} и \emph{Working Drafts} --- экспериментальные/переходные документы, которые могут затем войти в стандарт.

\subsubsection{Что такое The Zero Overhead Principle? Привести пример появления этого принципа в коде.}
\textbf{Определение:} \emph{Zero Overhead Principle} означает, что абстракции языка/библиотеки не должны навязывать прирост стоимости (время/память), которого нельзя избежать при реализации на более низком уровне. Иначе говоря, при разумном использовании абстракций вы должны получать производительность, сопоставимую с ручной реализацией.

\textbf{Последствия:}
\begin{itemize}
  \item Абстракции должны быть проектированы так, чтобы компилятор мог устранять накладные расходы (инлайнинг, оптимизация шаблонов, \texttt{constexpr}).
  \item Если накладные расходы неизбежны, должна быть возможность выбрать более низкоуровневый механизм.
\end{itemize}

\textbf{Пример:}
\begin{verbatim}
template <typename T>
inline T add(T a, T b) { return a + b; }

int main() {
    int x = add(1, 2); // оптимизатор может полностью убрать вызов
}
\end{verbatim}

Другой пример: итераторы STL для \texttt{std::vector<T>} при оптимизации компилируются в адресную арифметику (аналог работы с указателями) без дополнительных накладных расходов.

\subsubsection{Какие основные директивы CMake/файлов проекта? Для чего нужна директива \texttt{add\_executable}? Каковы её параметры?}
\textbf{Коротко о CMake:} CMake --- система генерации файлов сборки (build system generator). Часто используемые команды:
\begin{itemize}
  \item \texttt{cmake\_minimum\_required(VERSION <x>)} --- минимальная версия CMake.
  \item \texttt{project(<name> [LANGUAGES ...])} --- объявление проекта.
  \item \texttt{add\_executable(<name> [sources...])} --- создание исполняемого таргета.
  \item \texttt{add\_library(<name> [STATIC|SHARED|INTERFACE] [sources...])} --- создание библиотеки.
  \item \texttt{target\_include\_directories(<target> ...)} --- пути для заголовков.
  \item \texttt{target\_link\_libraries(<target> ...)} --- связывание с библиотеками.
  \item \texttt{find\_package(...)} --- поиск внешних пакетов.
  \item \texttt{target\_compile\_features}, \texttt{target\_compile\_options} --- установка флагов компиляции.
  \item \texttt{set(...)} --- установка переменных.
\end{itemize}

\textbf{add\_executable:} объявляет исполняемый таргет. Простейший синтаксис:
\begin{verbatim}
add_executable(<name> [WIN32] [MACOSX_BUNDLE] [EXCLUDE_FROM_ALL]
               source1 source2 ...)
\end{verbatim}
\textbf{Параметры:}
\begin{itemize}
  \item \texttt{<name>} --- имя таргета.
  \item список исходников --- имена файлов, которые войдут в таргет.
  \item опциональные флаги: \texttt{WIN32}, \texttt{MACOSX\_BUNDLE}, \texttt{EXCLUDE\_FROM\_ALL}.
\end{itemize}

\textbf{Пример:}
\begin{verbatim}
cmake_minimum_required(VERSION 3.15)
project(myprog LANGUAGES CXX)

add_executable(myprog
    src/main.cpp
    src/foo.cpp
)

target_include_directories(myprog PRIVATE include)
target_compile_features(myprog PRIVATE cxx_std_17)
\end{verbatim}

\textbf{Замечание:} в современном CMake рекомендуется работать через \texttt{target\_*} интерфейсы и указывать видимость зависимостей: \texttt{PRIVATE}, \texttt{PUBLIC}, \texttt{INTERFACE}. Это управляет тем, какие флаги/директории видимы только для самого таргета и какие транзитивно передаются потребителям.

\subsubsection{Что такое интерфейс и что такое реализация? Что из них идеальное, что материальное?}
\textbf{Интерфейс} --- формальное описание того, какие операции доступны внешнему коду: сигнатуры функций, объявленные типы, контракты поведения. Интерфейс отвечает на вопрос \emph{что} делать.

\textbf{Реализация} --- конкретный код, структуры данных и алгоритмы, которые исполняют интерфейс; отвечает на вопрос \emph{как} делать.

Интерфейс чаще ассоциируется с идеальной спецификацией (идеальное), а реализация --- с материальной частью (код, файлы, бинарники).

\subsubsection{Примеры разделения на интерфейс и реализацию в C++}
\textbf{Классический header + source:}
\begin{verbatim}
// include/foo.h  (интерфейс)
#pragma once
int add(int a, int b);
\end{verbatim}
\begin{verbatim}
// src/foo.cpp (реализация)
#include "foo.h"
int add(int a, int b) { return a + b; }
\end{verbatim}

\textbf{Модули C++20 (пример):}
\begin{verbatim}
// math.ixx  (модульный интерфейс)
export module math;
export int add(int a, int b);
\end{verbatim}
\begin{verbatim}
// math_impl.cpp  (модульная реализация)
module math;
int add(int a, int b) { return a + b; }
\end{verbatim}

В первых примерах интерфейс --- заголовочный файл, в модулях --- module interface unit (экспортируемая часть).

\subsubsection{Для чего используются заголовочные файлы в C++? Что в них обычно помещают? Что с ними обычно делают и что с ними нельзя? Какие расширения встречаются? Какова связь между заголовочными файлами и принципом интерфейс--реализация?}
\textbf{Назначение:} заголовочные файлы (headers) служат для объявления интерфейса: декларации функций, классов, шаблонов, типов и констант.

\textbf{Обычно помещают:}
\begin{itemize}
  \item объявления типов и классов (обычно лишь публичную часть),
  \item прототипы функций,
  \item шаблоны (полностью),
  \item inline/constexpr функции (которые должны быть видимы при инстанциации),
  \item using/typedef, enum, макросы (по необходимости).
\end{itemize}

\textbf{Чего обычно не кладут:}
\begin{itemize}
  \item большие реализации (если не inline/шаблон),
  \item определения глобальных переменных (вместо этого используют \texttt{extern} в заголовке и единственное определение в \texttt{.cpp}).
\end{itemize}

\textbf{Расширения:} \texttt{.h}, \texttt{.hpp}, \texttt{.hh}, \texttt{.hxx} --- проектная конвенция.

\textbf{Связь с интерфейсом/реализацией:} заголовочный файл обычно содержит интерфейсную часть; реализация хранится в .cpp (или в модуле реализации). Это реализует принцип разделения обязанностей и инкапсуляцию.

\subsubsection{Для чего используются модули трансляции в C++? Что в них обычно помещают? Что с ними делают и чего нельзя? Какие расширения встречаются? Связь с интерфейс-реализацией?}
\textbf{Назначение модулей:}
\begin{itemize}
  \item устранение недостатков препроцессорной модели \texttt{\#include} (меньше дублированных зависимостей, быстрее компиляция),
  \item явное разделение интерфейса (export) и реализации,
  \item более строгая и быстрая модель управления зависимостями.
\end{itemize}

\textbf{Обычно помещают:}
\begin{itemize}
  \item экспортируемые объявления в module interface unit,
  \item приватные реализации в модульных реализациях или в той же единице, но без \texttt{export}.
\end{itemize}

\textbf{Чего избегают / ограничения:}
\begin{itemize}
  \item не все макросы и\slash или препроцессорные приёмы совместимы с модульной организацией (модули и препроцессор взаимодействуют иначе),
  \item практика и детали зависят от компилятора; сборка модулей добавляет артефакты (BMI --- binary module interface) и порядок сборки становится важен.
\end{itemize}

\textbf{Расширения (конвенции):} \texttt{.ixx}, \texttt{.cppm}, \texttt{.mpp} и т.п. --- соглашения сообщества/компиляторов.

\textbf{Связь с интерфейсом/реализацией:} модули предоставляют современный механизм явного разделения интерфейса и реализации, часто с лучшими гарантиями по времени сборки и изоляции символов.

\bigskip
\textbf{Дополнительно: Этапы построения программы (кратко)}\\
Этапы:
\begin{enumerate}
  \item \textbf{Препроцессинг} (\texttt{\#include}, \texttt{\#define}, условная компиляция) --- текстовые преобразования (для классических includes).
  \item \textbf{Компиляция} --- перевод единицы трансляции в объектный код (парсинг, семантика, оптимизации).
  \item \textbf{Ассемблирование} --- если необходима отдельная стадия.
  \item \textbf{Линковка} --- связывание объектных модулей и библиотек, формирование исполняемого файла или библиотеки.
\end{enumerate}

\bigskip
\textbf{Правила именования файлов (Правило \#0) и договорённости по стилю оформления кода}

\begin{itemize}
  \item Принцип: согласованность важнее выбора конкретного правила.
  \item Общие рекомендации:
    \begin{itemize}
      \item исходники: \texttt{.cpp}, заголовки: \texttt{.h} / \texttt{.hpp};
      \item имена файлов маленькими буквами, разделение слов подчёркиванием или дефисом по соглашению проекта;
      \item структура каталогов должна отражать неймспейсы/модули;
      \item header-guard'ы формируются с учётом полного пути и имени проекта, например \texttt{PROJECT\_DIR\_SUBDIR\_FILENAME\_H}.
    \end{itemize}
  \item Стилевые договорённости:
    \begin{itemize}
      \item единый стиль отступов/имен (camelCase или snake\_case);
      \item интерфейс отделяется от реализации;
      \item минимизация области видимости (anonymous namespaces, internal linkage);
      \item документация контрактов интерфейса (пре/постусловия).
    \end{itemize}
\end{itemize}
